## 文档说明

本文档定义 CatchLight（拾光）照片查看与管理软件的功能需求、非功能需求、数据存储策略及用户交互规范。所有需求项均分配唯一编号，格式为 `FR-xxx`（功能需求）、`NFR-xxx`（非功能需求）、`DSR-xxx`（数据存储需求）、`FMT-xxx`（格式支持）、`UX-xxx`（用户交互需求）。

**优先级定义**：

| 优先级 | 含义 | 交付阶段 |
|--------|------|----------|
| **P0** | 必须实现，MVP 核心功能 | Phase 1 |
| **P1** | 重要功能，产品差异化能力 | Phase 2 |
| **P2** | 增强体验，对标 macOS 照片完整能力 | Phase 3 |
| **P3** | 可选功能，后期迭代 | Phase 4+ |

## 1. 产品愿景与定位

### 1.1 一句话定义

**CatchLight（拾光）是一款跨平台、本地优先的照片查看与管理软件——不复制、不导入，以 Everything 级极速索引就地浏览用户现有文件夹中的照片与视频，并提供去重、截图识别等智能整理能力。**

### 1.2 目标用户画像

| 用户类型 | 特征描述 | 核心诉求 |
|----------|----------|----------|
| **数码摄影爱好者** | 拥有大量 RAW/JPEG 文件，分散在多个硬盘/分区 | 快速浏览、按时间/地点/镜头组织，不破坏现有目录结构 |
| **多设备截图用户** | Windows/Linux 桌面用户，截图与照片混存 | 自动识别截图、去重、分类（代码/聊天/文档） |
| **跨平台用户** | 同时使用 Windows 与 Linux，拒绝 macOS 生态锁定 | 统一 UI/体验，轻量安装包，数据完全本地 |
| **隐私敏感用户** | 不希望照片上传云端 | 全部本地处理，无网络依赖（地图瓦片可选离线） |
| **大型图库用户** | 10 万+ 照片/视频 | 60fps 网格滚动、秒级搜索、快速首次索引 |

### 1.3 与竞品的定位差异

| 维度 | macOS 照片 | digiKam | iPhotron | Lap | **CatchLight（拾光）** |
|------|-----------|---------|----------|-----|------------------------|
| **平台** | 仅 macOS/iOS | Win/Mac/Linux | Win/Mac/Linux | 跨平台 | **Win + Linux** |
| **文件策略** | 导入到图库包，可能复制 | 导入数据库 | 就地索引，不复制 | 文件夹优先 | **就地索引，不复制** |
| **索引速度** | 系统级 Photos Library | 常规扫描 | 常规 walkdir 扫描 | 大库优化 | **MFT/USN 极速索引（Win）** |
| **UI 风格** | 简洁时间线 | 专业复杂 | macOS 风格 | 实用主义 | **macOS 照片风格** |
| **去重/截图识别** | 基础重复检测 | 无原生支持 | 不支持 | 不支持 | **三级去重 + 截图 AI 识别** |
| **技术栈** | 闭源 Swift | C++/Qt | Python/PySide6 | — | **Rust/Tauri，轻量高性能** |
| **导入/导出** | 核心工作流 | 核心工作流 | 无 | 无 | **明确不做导入/导出** |

**核心差异化总结**：

1. **不做导入/导出**：用户文件夹即图库，零迁移成本。
2. **极速索引**：Windows NTFS 上利用 MFT + USN Journal，对标 Everything/WizTree 性能。
3. **智能整理**：精确去重、感知去重、相似分组、截图识别——现有开源方案普遍缺失。
4. **跨平台 + 轻量**：Tauri 安装包 < 50MB，内存占用低于 Electron 方案。

## 2. 功能需求

### 2.1 核心浏览功能（P0）

#### 2.1.1 监控文件夹管理

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| **FR-001** | 用户可通过「设置 → 监控文件夹」或拖拽文件夹到侧边栏添加监控目录 | 添加后 5 秒内开始索引；UI 显示该目录及子项数量 |
| **FR-002** | 每个监控文件夹可独立设置「是否递归扫描子目录」 | 递归开关默认开启；关闭后仅索引该目录直接子文件，不进入子文件夹 |
| **FR-003** | 用户可删除已添加的监控文件夹 | 删除时弹出确认对话框，说明「仅从索引移除，不删除磁盘文件」；确认后从 `watched_folders` 表移除并清理该路径下文件的索引记录 |
| **FR-004** | 监控文件夹列表展示：路径、递归状态、文件数量、上次扫描时间 | 列表按添加时间排序；路径过长时中间省略显示 |
| **FR-005** | 支持添加多个不重叠或重叠的监控目录（重叠路径去重） | 子目录已被父目录递归覆盖时，添加子目录给出提示「已被上级目录覆盖」 |
| **FR-006** | 首次添加监控文件夹触发全量索引；后续依赖增量监听 | 全量索引在后台异步执行，不阻塞 UI |
| **FR-007** | 网络驱动器/NAS 路径可添加但标记为「慢速源」 | 慢速源降低扫描频率（默认 30 分钟轮询一次），UI 显示警告图标 |

#### 2.1.3 时间线分组浏览

### 2.2 快速索引引擎（P0）

#### 2.2.1 Windows 索引

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| **FR-100** | NTFS 分区上使用 **MFT 全量扫描** 作为首次索引方式 | 10 万媒体文件首次 MFT 扫描 < 10 秒（不含 EXIF 提取） |
| **FR-101** | MFT 扫描后按扩展名过滤，仅保留图片/视频格式（见 FMT 章节） | 过滤在 Rust 侧完成，不写入无关文件 |
| **FR-102** | 后续变更通过 **USN Journal** 增量监听感知 create/delete/rename/modify | 变更到索引更新延迟 < 1 秒 |
| **FR-103** | MFT/USN 访问需要管理员权限时，首次启动引导用户授权或提供「普通扫描」降级选项 | 降级提示说明性能差异；用户选择持久化 |
| **FR-104** | 非 NTFS 分区（exFAT/FAT32）降级为 `ReadDirectoryChangesW` + 递归扫描 | 功能正确，性能可接受（10 万文件 < 60 秒） |
| **FR-105** | USN Journal 缓冲区溢出或长期未运行时，自动触发增量全量比对 | 比对策略见 FR-110 |

#### 2.2.2 Linux 索引

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| **FR-110** | 首次索引使用 `walkdir` 递归扫描监控目录 | 10 万文件 < 30 秒（本地 SSD） |
| **FR-111** | 增量监听优先使用 **inotify**（`notify` crate） | 监听 create/delete/modify/rename 事件 |
| **FR-112** | inotify 监听数量超出系统上限时，提示用户增大 `fs.inotify.max_user_watches` 并降级为定期轮询 | 文档提供各发行版 sysctl 配置说明 |
| **FR-113** | 可选 fanotify 模式（需 root），作为高级设置 | 默认不启用 |
| **FR-114** | 支持 ext4/btrfs/xfs 等主流 Linux 文件系统 | CI 覆盖 Ubuntu 22.04+ |

#### 2.2.3 通用降级与增量策略

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| **FR-120** | 跨平台统一 `FileIndexer` trait 抽象：`full_scan()`、`watch_changes()` | Windows/Linux/Generic 三套实现，上层业务无感知 |
| **FR-121** | 通用降级方案：`std::fs` 递归扫描 + `notify` crate 文件监听 | 所有平台均可回退到此方案 |
| **FR-122** | 增量扫描策略：对比 **文件大小 + mtime**，任一变化则重新索引该文件 | 变更文件触发 EXIF 重提取 + 缩略图失效 |
| **FR-123** | 文件删除：索引中标记移除（非物理删除记录，最近删除功能见 FR-200） | 删除感知延迟 < 2 秒 |
| **FR-124** | 文件重命名/移动：更新 `file_path`，保留已有元数据（通过 inode/file_id 或路径映射） | Windows 使用 file reference number；Linux 使用 inode |
| **FR-125** | 扫描进度实时反馈：已扫描数/总数、当前路径、预估剩余时间 | UI 状态栏及「索引进度」面板可查看；支持后台最小化 |

#### 2.2.4 元数据提取流水线

| 编号 | 需求描述 | 验收标准 |
|------|----------|----------|
| **FR-130** | 索引完成后，EXIF 元数据提取在独立异步队列中执行，不阻塞文件枚举 | 并发度可配置（默认 CPU 核心数） |
| **FR-131** | 常见格式（JPEG/PNG/HEIF/TIFF）使用 Rust 原生库（`kamadak-exif`）解析 | 单张 EXIF 提取 < 5ms（JPEG） |
| **FR-132** | 特殊格式（部分 RAW/PSD）回退 ExifTool 子进程调用 | ExifTool 不可用时标记 `exif_extracted=failed` |
| **FR-133** | 提取结果写入 SQLite `media_files` 表及 FTS5 索引 | 提取完成后触发 UI 增量刷新 |

## 4. 数据存储需求

### 4.1 存储内容概览

| 数据类型 | 内容 | 可重建 | 默认位置 |
|----------|------|--------|----------|
| **索引数据库** | 文件路径、EXIF、哈希、分组、相簿关联 | ✅ 可重建 | 见方案对比 |
| **缩略图缓存** | micro/small/large WebP | ✅ 可重建 | 见方案对比 |
| **用户配置** | 监控目录、主题、语言、窗口状态 | ❌ 需备份 | 见方案对比 |
| **相簿 manifest** | 用户相簿元数据 | ⚠️ 部分可重建 | 见方案对比 |
| **Sidecar 编辑** | 非破坏性编辑参数 | ❌ 用户数据 | 与原图同目录 |
| **AI 模型** | CLIP/InsightFace ONNX | ✅ 可重新下载 | `~/.catchlight/models/` |
| **日志** | 运行日志 | ✅ 可删除 | `~/.catchlight/logs/` |
| **回收站** | Linux 删除文件副本 | — | `~/.catchlight/trash/` |

### 4.2 三种存储方案对比

#### 方案 A：集中存储

所有索引、缓存、配置存储于固定全局目录。

```
~/.catchlight/                    # Linux
%APPDATA%/CatchLight/             # Windows
├── config.toml                   # 用户配置
├── library.db                    # SQLite 主数据库
├── cache/
│   └── thumbs/
│       ├── micro/
│       ├── small/
│       └── large/
├── models/                       # AI 模型
├── logs/
├── trash/                        # Linux 回收站
└── backups/
```

| 优点 | 缺点 |
|------|------|
| 路径 predictable，备份简单 | 多盘符照片库时缓存/I/O 可能跨盘 |
| 卸载/迁移只需复制一个目录 | 用户不知道哪些目录被索引时不易理解 |
| 符合桌面应用惯例（Lightroom、digiKam） | 多用户共享照片目录时索引不跟随 |

#### 方案 B：分散存储

每张照片库目录下创建 `.catchlight/` 隐藏子目录，索引与缓存跟随照片位置。

```
D:/Photos/
├── 2024/
│   ├── IMG_001.jpg
│   └── .catchlight/
│       ├── index.db              # 该目录子索引
│       └── cache/thumbs/...
└── .catchlight/                  # 根级 manifest
    └── library.manifest.json     # 汇总引用
```

| 优点 | 缺点 |
|------|------|
| 索引跟随照片，移动硬盘即带走索引 | 全局搜索需聚合多个 DB，复杂度高 |
| 多盘符天然并行 I/O | 相簿跨目录关联困难 |
| 用户删除目录即清除对应索引 | 隐藏文件夹可能被误删/同步工具干扰 |
| — | 同一文件被多目录监控时可能重复索引 |

#### 方案 C：混合方案（推荐）

**全局索引集中 + 相簿 manifest 分散/集中 + 缩略图集中缓存**

```
~/.catchlight/                    # 全局集中
├── config.toml
├── library.db                    # 统一 SQLite（所有监控目录）
├── cache/thumbs/                 # 缩略图集中（或按 hash 分片）
├── models/
├── logs/
└── albums/                       # 用户相簿 manifest（JSON 导出备份）
    └── user-albums.json

# 可选：sidecar 编辑文件跟随原图
D:/Photos/IMG_001.jpg
D:/Photos/IMG_001.jpg.catchlight.json
```

| 优点 | 缺点 |
|------|------|
| 全局 FTS5 搜索、去重、时间线简单高效 | 移动照片库后需更新路径（通过 USN/inotify 感知） |
| 缩略图集中管理，LRU 缓存统一 | 缓存目录可能较大 |
| 相簿 manifest 可导出 JSON 便于备份 | 比纯方案 A 略复杂 |
| 兼顾性能与可维护性 | — |

### 4.3 推荐方案及理由

**推荐采用方案 C（混合方案）**，理由：

1. **搜索与去重**：10 万+ 库的 FTS5 全文搜索、BLAKE3 去重、相似度计算依赖全局视图，单一 SQLite 数据库最高效。
2. **索引性能**：MFT/USN 扫描结果直接写入 central DB，无需合并多库。
3. **缓存效率**：缩略图按 BLAKE3 hash 命名，集中存储天然去重（同一文件多处硬链接仅一份缓存）。
4. **用户迁移**：`~/.catchlight/` 整体复制即可迁移全部配置与索引；重建索引亦可完全恢复。
5. **Sidecar 分散**：编辑参数跟随原文件，符合非破坏性编辑语义，不增加全局 DB 负担。

**路径规范（方案 C 最终定义）**：

| 平台 | 全局目录 |
|------|----------|
| Windows | `%APPDATA%\CatchLight\` |
| Linux | `$XDG_DATA_HOME/catchlight/` 或 `~/.catchlight/` |

**DSR-001**：应用首次启动时创建全局目录结构，若不存在则初始化空数据库。  
**DSR-002**：缩略图缓存文件名 `{blake3_hex_prefix_16}.webp`，三级分目录存储。  
**DSR-003**：用户可在设置中自定义全局目录路径（需重启生效）。  
**DSR-004**：「存储管理」页面展示：数据库大小、缓存大小、各监控目录文件数；提供「清空缓存」「重建索引」按钮。  
**DSR-005**：Sidecar 文件格式：

```json
{
  "version": 1,
  "original_path": "/path/to/IMG_001.jpg",
  "edits": {
    "crop": { "x": 100, "y": 50, "width": 800, "height": 600 },
    "rotation": 90,
    "flip_horizontal": false,
    "flip_vertical": false
  },
  "updated_at": "2026-06-28T12:00:00Z"
}
```

### 5.2 RAW 格式

| 编号 | 格式 | 扩展名 | 支持策略 |
|------|------|--------|----------|
| **FMT-020** | Canon | `.cr2`, `.cr3` | 嵌入 JPEG 预览 + ExifTool 元数据 |
| **FMT-021** | Nikon | `.nef`, `.nrw` | 同上 |
| **FMT-022** | Sony | `.arw` | 同上 |
| **FMT-023** | Adobe | `.dng` | 同上；DNG 优先原生解析 |
| **FMT-024** | Fujifilm | `.raf` | 同上 |
| **FMT-025** | Olympus | `.orf` | 同上 |
| **FMT-026** | Panasonic | `.rw2` | 同上 |
| **FMT-027** | Pentax | `.pef` | 同上 |
| **FMT-028** | Leica | `.rwl`, `.dng` | 同上 |
| **FMT-029** | 其他 RAW | `.raw`, `.3fr`, `.srw` 等 | ExifTool 兜底；无预览则显示占位 |

**RAW 支持策略（FMT-030）**：

1. **索引阶段**：识别扩展名，提取 ExifTool 元数据（相机、镜头、拍摄时间）。
2. **预览阶段**：优先提取 RAW 内嵌 JPEG 预览生成缩略图及 large 预览。
3. **查看阶段**：显示嵌入预览；无嵌入预览时显示「请使用外部 RAW 编辑器」。
4. **不做 RAW 显影**：CatchLight 不是 RAW 编辑器，不提供完整 demosaic 渲染（P3 以后不计划）。
